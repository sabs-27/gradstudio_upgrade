    Data plan

    Where every kind of information lives

    The biggest beginner mistake is mixing data types. GradStudio stays clean because relational data, object files, fast config, and browser-only state each have a proper home.

    D1 database

    Use D1 when records need relationships, sorting, querying, ownership, or reporting.

    courses sections simulations users user_pins comments analytics

    R2 bucket

    Use R2 when the item is a file: HTML lessons, images, generated assets, static CSS, JavaScript, icons, and admin uploads.

    static/ dev-static/ carousel/images/ simulations/

    KV namespaces

    Use KV for small values that must be read quickly at the edge and do not need relational joins.

    learn carousel config refresh tokens rate limits

    Browser storage

    Use local storage only as a convenience cache. The account-backed version of important state belongs in D1.

    auth token cache temporary pins UI preferences

    Worker memory

    Use Worker runtime code for normalization, validation, response shaping, route matching, and security headers.

    sanitize authorize rate limit normalize

    Git repository

    Use Git for source files only. Keep previews, scratch folders, browser profiles, and generated junk out of tracked source.

    static/ worker/src/ worker/migrations/ deploy scripts
    File map

    The files a new builder should know first

    You do not need to memorize every file. Start with these source areas and learn what responsibility each owns.

    Frontend files

    • static/index.html defines the main production shell.
    • static/gradstudio-learn-demo.css owns the current learner visual system.
    • static/gradstudio-learn-demo.js renders courses, auth, pins, carousel blocks, and page states.
    • static/gradstudio-learn-admin.html and static/gradstudio-learn-admin.js power the modern admin editor.
    • static/architecture.html documents the existing architecture at a high level.

    Worker and storage files

    • worker/src/index.js is the central API router and static file server.
    • worker/src/handlers/ contains focused admin, analytics, upload, replace, sync, and carousel handlers.
    • worker/src/middleware/ contains auth, authorization, CORS, rate limit, and error wrappers.
    • worker/src/utils/ contains auth helpers, sanitizer, validators, security logging, and response helpers.
    • worker/wrangler.toml defines Cloudflare routes, D1, R2, KV bindings, and environment variables.
    Deploy flow

    How code becomes the live site

    Deployment is two jobs: upload browser assets to R2, then deploy the Worker that serves those assets and APIs. Keep dev prefixes away from production prefixes.

    1
    Edit source locallyChange files under static/ and worker/src/. Avoid editing generated previews as source.
    2
    Run local checksOpen the HTML, test routes, inspect console errors, and verify mobile and desktop layouts.
    3
    Upload static assetsPut HTML, CSS, JS, images, and lesson files into R2 under static/ or dev-static/.
    4
    Deploy WorkerWrangler deploys the edge API and static file routing defined by worker/wrangler.toml.
    5
    Smoke test live URLsCheck homepage, course APIs, auth flow, carousel config, admin editor, and a sample lesson.
    Practical rule: every production change should answer three questions before deploy: what changed in the browser, what changed in the Worker, and which storage system receives the new data or files?
    Final checklist

    How to know the build is complete

    This checklist turns the diagram into a working definition of done.

    Frontend done

    • Homepage loads without console errors.
    • Catalog renders from API data, not hardcoded-only data.
    • Course detail and player states are reachable.
    • Mobile layout does not overlap or clip controls.

    API done

    • Public routes return JSON with security headers.
    • Auth routes issue and refresh valid tokens.
    • Admin routes validate input before writing.
    • Errors return useful status codes and safe messages.

    Storage done

    • D1 has required tables and indexes.
    • R2 contains the expected static and upload prefixes.
    • KV keys are environment-specific where needed.
    • Browser cache is treated as optional, not primary data.

    Admin done

    • Admin can create folders, courses, sections, and simulations.
    • Uploads land in R2 and create matching D1 records.
    • Carousel edits save and appear on the learner homepage.
    • Recycle, restore, and permanent delete flows are deliberate.

    Security done

    • CORS only allows intended origins.
    • Rate limits protect public, auth, user, and admin routes.
    • HTML uploads are scanned or wrapped before serving.
    • Role checks protect sensitive admin actions.

    Deploy done

    • Dev and production prefixes stay separate.
    • Static assets include cache-busting when needed.
    • Worker bindings match the target environment.
    • Live smoke tests pass after deployment.
